## 테라폼 학습 로드맵

  목차 1: 테라폼 기초 (The Core)

  이 단계에서는 Terraform의 가장 기본적인 개념과 핵심 워크플로우를 이해하는 데 집중합니다.

   1. IaC(Infrastructure as Code)와 테라폼의 이해
       * IaC란 무엇이며 왜 필요한가?
       * 테라폼의 특징: 선언적 언어, 에이전트리스 방식, 멀티 클라우드 지원
   2. HCL (HashiCorp Configuration Language) 기본 문법
       * 테라폼 코드의 기본 구조 (<BLOCK_TYPE> "<BLOCK_LABEL>" "<BLOCK_LABEL>" { ... })
       * 주석, 문자열, 숫자, 불리언 등 기본 데이터 타입
   3. 프로바이더 (Providers)
       * 프로바이더의 역할: Terraform과 AWS, Azure, Proxmox 등 외부 API를 연결하는 플러그인
       * required_providers 블록을 사용하여 프로바이더 선언 및 버전 고정하기
   4. 리소스 (Resources)
       * 리소스의 개념: 인프라의 가장 기본적인 단위 (예: AWS EC2 인스턴스, Proxmox VM)
       * resource 블록을 사용하여 인프라 리소스 정의하기
   5. 핵심 워크플로우 (The Core Workflow)
       * terraform init: 작업 환경 초기화 및 프로바이더 다운로드
       * terraform plan: 실행 계획 미리보기 (가장 중요한 명령어 중 하나)
       * terraform apply: 실행 계획을 실제 인프라에 적용
       * terraform destroy: 테라폼으로 생성한 모든 리소스 삭제

  > 학습 목표: 간단한 리소스(예: Proxmox VM 1대 또는 AWS S3 버킷 1개)를 코드로 생성하고 삭제할 수 있게 됩니다.

  목차 2: 코드 구조화 및 동적 구성

  하드코딩된 정적인 코드를 넘어, 재사용 가능하고 유연한 코드를 작성하는 방법을 배웁니다.

   1. 변수 (Variables)
       * 입력 변수 (`variable`): 코드의 재사용성을 높이기 위해 값을 외부에서 주입하는 방법
       * 출력 변수 (`output`): 생성된 리소스의 정보(예: VM의 IP 주소)를 외부로 내보내는 방법
       * 변수 값 전달 방법: .tfvars 파일, -var 커맨드라인 플래그, 환경 변수
   2. 데이터 소스 (Data Sources)
       * data 블록의 역할: 코드로 생성하지 않은 기존 리소스의 정보를 읽어오는 기능
       * 예시: 최신 Ubuntu AMI ID 가져오기, 기존에 생성된 VPC 정보 참조하기
   3. 지역 변수 (Local Values)
       * locals 블록을 사용하여 코드 내에서 사용할 임시 변수나 복잡한 표현식을 단순화하기
   4. 표현식과 내장 함수 (Expressions & Built-in Functions)
       * Meta-Argument: count (동일한 리소스 여러 개 생성), for_each (맵이나 세트를 기반으로 각각 다른 속성의 리소스 생성)
       * 조건문: count와 삼항 연산자(condition ? true_val : false_val)를 이용한 조건부 리소스 생성
       * 내장 함수: length(), lookup(), cidrsubnet(), format() 등 다양한 내장 함수 활용법

  > 학습 목표: 변수를 활용하여 재사용 가능한 코드를 작성하고, count나 for_each를 사용해 여러 개의 리소스를 동적으로 관리할 수 있게 됩니다.

  목차 3: 상태(State) 관리

  Terraform이 실제 인프라와 코드를 어떻게 연결하는지 이해하는, 협업을 위한 가장 중요한 파트입니다.

   1. 테라폼 상태 파일 (`terraform.tfstate`)
       * 상태 파일의 역할: 코드와 실제 인프라 리소스 간의 매핑 정보 저장
       * 왜 상태 파일이 중요한가? (Terraform의 두뇌)
   2. 로컬 상태(Local State) vs 원격 상태(Remote State)
       * 팀원과 협업 시 로컬 상태 파일의 문제점
       * 원격 상태 저장의 필요성
   3. 백엔드 (Backends)
       * 백엔드의 개념: 상태 파일을 원격에 저장하기 위한 설정
       * 자주 사용되는 백엔드: AWS S3, Azure Blob Storage, Terraform Cloud
   4. 상태 잠금 (State Locking)
       * 동시에 여러 사람이 apply 하는 것을 방지하여 상태 파일이 깨지는 것을 막는 기능
       * S3 백엔드 사용 시 DynamoDB를 이용한 상태 잠금 설정 방법

  > 학습 목표: 팀 프로젝트에서 안전하게 협업할 수 있도록 상태 파일을 원격(S3)으로 관리하고 잠금을 설정할 수 있게 됩니다.

  목차 4: 코드 재사용 및 확장 (모듈)

  대규모 인프라를 관리하기 위한 필수 개념입니다.

   1. 모듈 (Modules)
       * 모듈의 개념: 관련 리소스들을 묶어놓은 재사용 가능한 코드 블록
       * 왜 모듈을 사용해야 하는가? (코드의 추상화, 재사용성, 유지보수 용이성)
   2. 모듈 만들기 및 사용하기
       * 로컬 경로에 직접 만든 모듈 호출하기
       * 모듈의 입력 변수(variable)와 출력(output) 설계
   3. 모듈 소스 (Module Sources)
       * Terraform 레지스트리(Registry)에 있는 공개 모듈 사용하기
       * Git 저장소에 있는 모듈 사용하기

  > 학습 목표: VPC, 서브넷, EC2 등 자주 사용하는 인프라 구성을 모듈로 만들어, 여러 프로젝트에서 쉽게 재사용할 수 있게 됩니다.

  ---

  목차 5: 고급 주제 및 생태계

  Terraform을 더 전문적으로 사용하고 CI/CD 파이프라인에 통합하는 방법을 배웁니다.

   1. 작업 공간 (Workspaces)
       * terraform workspace를 사용하여 동일한 코드로 개발(dev), 스테이징(stg), 운영(prod) 환경을 분리하여 관리하는 방법
   2. 프로비저너 (Provisioners)
       * 리소스 생성 후 특정 스크립트를 실행하는 기능 (remote-exec, local-exec)
       * 주의: 프로비저너는 최후의 수단이며, 설정 관리에는 Ansible, Packer 같은 전문 도구를 사용하는 것이 권장됨
   3. CI/CD 파이프라인 연동
       * GitHub Actions, Jenkins 등에서 terraform plan, apply를 자동화하는 방법
       * PR(Pull Request) 생성 시 plan 결과를 댓글로 남겨 코드 리뷰하기

  이 로드맵에 따라 꾸준히 프로젝트를 진행하며 학습하신다면, 어떤 클라우드 환경에서도 인프라를 능숙하게 코드로 관리하는 전문가가 되실 수 있을 겁니다.
