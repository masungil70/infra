## 4. 네트웩 구성도

위 시스템을 사용하여 Ceph 저장소 및 Client 1 ~ 16대를 각 4대씩 묶어서 클러스터를 구성시 네트웩 구성도

이유 : Ceph 용 네트웍을 구성은 spine-leaf 구조로 이중화, 10G을 2개 묶은 LACP를 활용할 수 있음

### **가장 좋은 구조 (성능 + 안정성)**

1. **물리 분리 + 링 구조 → ❌ 비추천 (병목 발생)**
2. **VLAN 분리 + Spine-Leaf 구조 → 현실적 최적**
3. **물리 분리 + Spine-Leaf 구조 → 최상**

---

### 🔥 Ceph 트래픽 특징

* Public: Client 요청 (상대적으로 적음)
* Cluster: OSD 간 복제 (엄청 많음, 폭발적)

👉 실제로:

```text
Write 1번 → 내부 복제 3번 발생
```

👉 즉,

* Cluster 네트워크가 **3~10배 더 바쁨**

---

### 그래서 물리 분리를 하면

| 장점    | 설명              |
| ----- | --------------- |
| 완전 격리 | 트래픽 간섭 없음       |
| 안정성   | broadcast 영향 없음 |
| 예측 가능 | 성능 일정           |

👉 그래서 “이론적으로 최고”

---

### 그런데 왜 Spine-Leaf가 더 중요해졌나

👉 이유: **네트워크 구조가 성능을 더 크게 좌우함**

---

### ❌ 링 구조 문제

* STP → 링크 차단
* 우회 경로 → latency 증가
* 대역폭 낭비

👉 결과:

```text
10G 네트워크 → 실제 3~5G 수준으로 내려감 
```

---

### ⭕ Spine-Leaf 구조

* 모든 Leaf → Core(Spine) 직접 연결
* 병렬 경로 존재
* ECMP / LACP 활용 가능

👉 결과:

```text
10G → 거의 9~10G 사용 가능
```

---

### 🥊 물리 분리 vs VLAN 분리

| 항목  | 물리 분리 | VLAN 분리 |
| --- | ----- | ------- |
| 성능  | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐    |
| 안정성 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐     |
| 비용  | ❌ 높음  | ⭕ 낮음    |
| 운영  | ❌ 복잡  | ⭕ 쉬움    |

---

### 🥊 구조 비교

| 구조                 | 성능      |
| ------------------ | ------- |
| 물리 분리 + 링          | ❌ 낮음    |
| VLAN + Spine-Leaf  | ⭕ 매우 높음 |
| 물리 분리 + Spine-Leaf | 🚀 최고   |

---


### 🎯 현실적인 최적 해답

👉 **Spine-Leaf + VLAN 분리**

```text
이유:
- 스위치 수 제한
- 포트 부족
- 운영 복잡도 증가 방지
```

---

### 🎯 최고 성능을 원하면

👉 **Public / Cluster 물리 분리 + Spine-Leaf 2세트**

```text
[Public Fabric]
Core 2대 + Leaf 2대

[Cluster Fabric]
Core 2대 + Leaf 3대
```

👉 단점:

* 스위치 2배 필요
* 비용 증가

---

### 왜 VLAN도 충분히 빠른가

요즘 스위치는:

* ASIC 기반 처리
* VLAN 태깅 = 거의 비용 없음

👉 실제 차이:

```text
물리 분리 vs VLAN
→ 5% 이내 차이
```

---

### ✔️ 반드시 물리 분리 해야 하는 경우

* 25G / 40G 이상 환경
* 수십 노드 이상
* 금융 / HPC / AI 학습

---

### ✔️ VLAN으로 충분한 경우

* 10G
* 20 노드 미만
* 일반 Ceph

---

### 가장 중요한 우선순위

👉 이 순서가 핵심입니다

```text
1순위: 네트워크 구조 (Spine-Leaf)
2순위: MTU 9000
3순위: LACP
4순위: 트래픽 분리 (VLAN or 물리)
```

### 10G 네트웩 구성도

👉 총 7대 구성:

* Spine: 2대
* Ceph Leaf: 3대
* Client Leaf: 2대

```
                  ┌──────────┐   (LACP)   ┌──────────┐
                  │  Spine1  │============│  Spine2  │
                  └─────┬────┘            └─────┬────┘
        ┌───────────────┼───────────────────────┼────────────┐
        ││              ││           ││         ││          ││
     Leaf-c1         Leaf-c2     Leaf-c3      Leaf-u1     Leaf-u2 
     (Ceph)          (Ceph)       (Ceph)      (Clinet)    (Clinet) 
     ││ ││ ││        ││ ││ ││     ││  ││      ││││││      ││││││  
     C1 C2 ││        C3 C4 ││     C5  C6      Clinet1~6   Clinet7~12
       Clinet13,14    Clinet15,16 
```

---

### Spine-Leaf 구조 장점

* 모든 트래픽이 **최단 경로**
* 병렬 전송 가능
* LACP로 대역폭 확장
* 장애 시 자동 우회

👉 Ceph에 최적 구조

---

### 물리 배치 설계

---

#### Spine 스위치 (2대)

👉 역할:

* 전체 트래픽 집선
* LACP Aggregation

#### 연결

```text
Spine1 ───── Spine2 (LACP 2포트)
```

---

#### Ceph Leaf (3대)

👉 각 Leaf에 Ceph Node 2대씩 연결

```text
Leaf1 → Ceph Node1, Ceph Node2
Leaf2 → Ceph Node3, Ceph Node4
Leaf3 → Ceph Node5, Ceph Node6
```

👉 각 Leaf는 Core 2대로 연결

```text
Leaf ─── Core1 (10G)
Leaf ─── Core2 (10G)
```

👉 = **Dual uplink (Active-Active)**

---

#### Client Leaf (2대)

```text
Leaf4 → PC 6대
Leaf5 → PC 6대

Leaf1 → PC 2대
Leaf2 → PC 2대

```
