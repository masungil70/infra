## Ceph(세프)를 Proxmox에 설지하기 

### 사전 조건 
1. nvme 디스크 500G, SATA 디스크 1T를 준비한다.
2. 10G NIC 2포트를 준비한다
3. ceph cluster 구성을 한 PC 4대

### 개요
- NVMe 디스크 전체를 사용하지 않고
- **OS 용도로 300G만 사용**
- 나머지 공간은 Ceph DB/WAL, DB 서버 등으로 활용

### 설치 방법 핵심
- Proxmox 설치 화면에서 **Options (Advanced Options)** 클릭
- `hdsize=300` 설정

### 결과 구조 예시
```
nvme0n1 (500G)
├─ EFI /boot
├─ pve-root
├─ pve-swap
├─ pve-data
└─ (미사용 영역 약 200G)
```

## 2. LVM 구조 개념 정리

### LVM 3단계 구조
| 단계 | 이름 | 설명 |
|----|----|----|
| 1 | PV (Physical Volume) | 실제 디스크/파티션 |
| 2 | VG (Volume Group) | PV 묶음 |
| 3 | LV (Logical Volume) | 실제 사용하는 논리 볼륨 |

Ceph DB/WAL은 **파일시스템 없이 Raw Block(LVM)** 형태로 사용.

---

## 3. NVMe 남은 영역을 Ceph 전용 VG로 구성

### 파티션 예

### VG 생성

## 4. Ceph BlueStore DB / WAL 개념

### BlueStore 내부 구조
```
OSD DATA (HDD)
└─ BlueStore
   ├─ block      (DATA)
   ├─ block.db   (Metadata, NVMe)
   └─ block.wal  (Write Ahead Log, NVMe)
```

### 권장 크기
| 항목 | 크기 |
|----|----|
| DB | 50G |
| WAL | 8G |

### 15. 네트웩트 인터페이스에 내부 IP를 설정
```bash
vi /etc/network/interfaces

auto lo
iface lo inet loopback

#외부 인터넷을 위한 설정 
iface nic0 inet manual

#외부 인터넷을 위한 설정 ip 설정 
auto vmbr0
iface vmbr0 inet static
        address 192.168.3.94/24
        gateway 192.168.3.1
        bridge-ports nic0
        bridge-stp off
        bridge-fd 0

#1번째 10G NIC 내부 통신을 위한 설정 ip 설정 
iface nic1 inet manual
        mtu 9000

auto vmbr1
iface vmbr1 inet static
        address 10.10.10.101/24
        bridge-ports nic1
        bridge-stp off
        bridge-fd 0
        mtu 9000

#2번째 10G NIC 내부 통신을 위한 설정 ip 설정 
iface nic2 inet manual
        mtu 9000

auto vmbr2
iface vmbr2 inet static
        address 10.10.20.101/24
        bridge-ports nic2
        bridge-stp off
        bridge-fd 0
        mtu 9000

source /etc/network/interfaces.d/*
```

### 31. Scrub 이란 무엇인가?

**Scrub = 데이터 정합성 검사**

Ceph는 디스크, 네트워크, 노드가 많기 때문에
시간이 지나면서 다음과 같은 문제가 생길 수 있습니다.

* 비트 로트(Bit rot)
* 디스크 조용한 오류(Silent corruption)
* 복제본 간 데이터 불일치
* 메타데이터 손상

👉 이를 **미리 발견하기 위해 주기적으로 검사**하는 것이 Scrub입니다.

**Scrub = 데이터 정합성 검사**

Ceph는 디스크, 네트워크, 노드가 많기 때문에
시간이 지나면서 다음과 같은 문제가 생길 수 있습니다.

* 비트 로트(Bit rot)
* 디스크 조용한 오류(Silent corruption)
* 복제본 간 데이터 불일치
* 메타데이터 손상

👉 이를 **미리 발견하기 위해 주기적으로 검사**하는 것이 Scrub입니다.

---

## 2️⃣ Scrub 종류

### 🔹 1. Scrub (Light Scrub)

* **메타데이터 위주 검사**
* 객체 존재 여부, 크기, 체크섬 등 확인
* 부하가 비교적 낮음

### 🔹 2. Deep Scrub

* **실제 데이터까지 읽어서 검사**
* 복제본 간 데이터 비교
* 디스크 I/O 부하 큼
* 시간이 오래 걸림

---

## 3️⃣ Scrub 시간 조정이 필요한 이유

Scrub은 **클러스터 성능에 직접적인 영향**을 줍니다.

### 🚨 기본 설정 그대로 두면

* 주간 / 주말 / 낮 시간에도 scrub 실행
* VM 디스크, DB I/O 성능 저하
* Ceph latency 증가
* 교육/실습 환경에서 체감 심함

### ✅ 그래서 보통

* **야간에만 수행**
* **업무 시간에는 중단 또는 최소화**
* **Deep scrub 주기 늘림**

을 합니다.

---

## 4️⃣ Scrub 시간 조정이란 무엇을 조정하는가?

### 📌 1. Scrub 허용 시간대

> 언제 Scrub을 실행할 수 있는가

* `osd_scrub_begin_hour`
* `osd_scrub_end_hour`

예시:

```bash
ceph config set osd osd_scrub_begin_hour 1
ceph config set osd osd_scrub_end_hour 7
```

➡️ **새벽 1시 ~ 7시**에만 scrub 허용

---

### 📌 2. Scrub 빈도 (주기)

| 항목                        | 의미            | 기본값(대략) |
| ------------------------- | ------------- | ------- |
| `osd_scrub_min_interval`  | 최소 주기         | 1일      |
| `osd_scrub_max_interval`  | 최대 지연         | 7일      |
| `osd_deep_scrub_interval` | Deep scrub 주기 | 7~14일   |

예시:

```bash
ceph config set osd osd_scrub_min_interval 172800   # 2일
ceph config set osd osd_scrub_max_interval 604800   # 7일
ceph config set osd osd_deep_scrub_interval 1209600 # 14일
```

---

### 📌 3. Scrub 동시성 / 강도 조절

| 설정                   | 설명                      |
| -------------------- | ----------------------- |
| `osd_max_scrubs`     | OSD당 동시에 수행 가능한 scrub 수 |
| `osd_scrub_sleep`    | 객체 처리 사이 대기시간           |
| `osd_scrub_priority` | scrub 우선순위              |

예시:

```bash
ceph config set osd osd_max_scrubs 1
ceph config set osd osd_scrub_sleep 0.1
```

➡️ **천천히, 덜 공격적으로 수행**

---

## 5️⃣ Scrub 시간 조정 안 하면 생기는 문제

| 상황            | 결과             |
| ------------- | -------------- |
| Scrub 미실행     | 데이터 손상 발견 못함   |
| 과도한 Scrub     | VM, DB 성능 급락   |
| 낮 시간 Scrub    | 교육/실습 체감 성능 저하 |
| Deep scrub 과다 | 디스크 수명 감소      |

👉 **적절한 시간 + 적절한 빈도**가 핵심

---

## 6️⃣ 실무 권장 설정 (교육/Proxmox + Ceph 환경)

💡 현재 구성(다수 VM, 1G/10G 혼합망, HDD OSD 기준)을 고려하면:

```bash
# Scrub 허용 시간: 새벽
ceph config set osd osd_scrub_begin_hour 1
ceph config set osd osd_scrub_end_hour 7

# 주기 완화
ceph config set osd osd_scrub_min_interval 259200     # 3일
ceph config set osd osd_scrub_max_interval 604800     # 7일
ceph config set osd osd_deep_scrub_interval 1209600   # 14일

# 부하 완화
ceph config set osd osd_max_scrubs 1
ceph config set osd osd_scrub_sleep 0.1
```
